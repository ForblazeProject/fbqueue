## 1. プロジェクトの哲学 (Core Philosophy)

FBQueue は、以下の原則に基づいて設計・実装されています。

*   **ゼロ依存 (Zero Dependency)**: Rust 標準ライブラリのみを使用し、外部クレート（`clap`, `serde`, `tokio` など）に一切依存しません。これにより、極めて高いポータビリティと軽量性を実現します。
*   **シングルバイナリ (Single Binary)**: コンパイルされた実行ファイルは一つのみです。どのような Linux または Windows 環境でも、バイナリをコピーするだけで動作します。
*   **透過性 (Transparency)**: すべてのジョブの状態とデータは、ファイルシステム上の分かりやすいディレクトリ構造に保存されます。ユーザーは OS 標準のコマンド (`ls`, `cat` など) でデーモンの内部状態を直接確認できます。
*   **オートオフ (Auto-Off Daemon)**: デーモンはジョブがないアイドル状態が 5 分続くと自動的に終了します。これにより、共用サーバでのリソースの無駄遣いを防ぎ、ユーザーフレンドリーな運用を可能にします。
*   **堅牢性 (Robustness)**: デーモンが予期せずクラッシュしても、ジョブの状態はファイルシステムに永続化されています。デーモン再起動時には、中断されたジョブを自動的に復旧（再投入）し、処理を続行します。
*   **ユーザー中心設計**: 計算科学の現場でのニーズ（大量ジョブの一括投入、リソースの重み付け、既存スクリプトの活用）を最優先します。

## 2. ディレクトリ構造 (Directory Structure)

FBQueue は、ユーザーのホームディレクトリ直下の隠しフォルダ `~/.fbqueue/` をデータ格納場所として使用します。

```
~/.fbqueue/
├── config                     # 全体の設定ファイル (capacityなど)
├── queue/                     # ジョブのキュー
│   ├── new/                   # 新規投入されたジョブファイル (未処理)
│   ├── running/               # 実行中のジョブファイル
│   ├── done/                  # 完了したジョブファイル
│   ├── failed/                # 失敗したジョブファイル
│   └── cancel/                # キャンセル待ちのジョブファイル (デーモンが検知して終了させる)
├── logs/                      # デフォルトのジョブログ出力先 (ジョブが-o/-eを指定しない場合)
└── run/                       # デーモン管理用ファイル
    ├── daemon.pid             # 実行中のデーモンの PID
    ├── last_id                # 次に発行するジョブID (カウンター)
    └── id.lock/               # ID発行時の排他制御用ロックディレクトリ
```

## 3. ジョブファイル形式 (.job)

`~/.fbqueue/queue/` ディレクトリ内の各ジョブは、テキストファイルとして保存されます。ファイル名はジョブ ID (`1.job`, `2.job` など) です。

```
id: 1                        # ジョブのユニークID (連番)
name: MySpecialJob           # ジョブ名 (-N/-J オプションまたはスクリプトから)
cmd: ./test_script.sh        # 実行コマンド
cost: 4                      # このジョブが消費するリソース量 (capacity の単位)
stdout: /path/to/stdout.txt  # 標準出力のリダイレクト先 (オプション)
stderr: /path/to/stderr.txt  # 標準エラー出力のリダイレクト先 (オプション)
arg: arg1                    # コマンドへの引数 (arg: 行は複数存在しうる)
arg: arg2
cwd: /home/user/project      # ジョブが実行されたカレントディレクトリ
env: PATH=/usr/bin:/bin      # 環境変数 (env: KEY=VALUE 形式で複数存在しうる)
env: MY_VAR=value
```
*   `stdout` と `stderr` はオプションです。指定がなければ PBS スタイルのデフォルト名で `job.cwd` に生成されます。
*   `env` は `fbqueue sub` 実行時の環境変数がすべて保存されます。これによりジョブ実行時の環境が完全に再現されます。

## 4. コマンドラインインターフェース (CLI)

`fbqueue` バイナリは、実行時の引数または `argv[0]` (コマンド名) に応じて動作を変えます。

*   **`fbqueue sub` / `qsub`**: ジョブをキューに投入します。
    *   **オプション**:
        *   `-c COST` / `--cost COST`: ジョブが消費するリソースコスト（デフォルト: 1）。
        *   `-N NAME` / `-J NAME`: ジョブの名前（デフォルト: コマンド名）。
        *   `-o OUTFILE`: 標準出力のリダイレクト先ファイル。
        *   `-e ERRFILE`: 標準エラー出力のリダイレクト先ファイル。
        *   `--range N-M`: `N` から `M` までの数値で `{}` プレースホルダを置換し、複数のジョブを一括投入。
        *   `--list A,B,C`: カンマ区切りの文字列で `{}` プレースホルダを置換し、複数のジョブを一括投入。
    *   **挙動**: ジョブファイルを作成し `~/.fbqueue/queue/new/` に置き、デーモンが停止していれば自動的にバックグラウンドで起動します。

*   **`fbqueue stat` / `qstat`**: 現在のジョブの状態を表示します。
    *   `Pending`, `Running` なジョブの ID, 名前, コストを表示。
    *   デーモンが使用中の `capacity` も表示。

*   **`fbqueue del JOB_ID` / `qdel JOB_ID`**: 指定されたジョブを削除またはキャンセルします。
    *   `new/` にあれば削除。
    *   `running/` にあれば、`cancel/` にシグナルファイルを置き、デーモンが検知してプロセスを終了させます。

*   **`fbqueue daemon <start|stop|status>`**: デーモンプロセスを直接管理します。
    *   `start`: デーモンを起動します。`ensure_daemon` が呼び出すのと同様のロジックです。
    *   `stop`: デーモンプロセスを終了させます。
    *   `status`: デーモンが動作中か、PID と共に表示します。

## 5. デーモン (Daemon) の動作

`fbqueue daemon start` によって起動されるデーモンは、以下のループを 1 秒ごとに実行します。

1.  **ロックファイルのチェック**: `~/.fbqueue/run/daemon.pid` をチェックし、自身以外にデーモンが実行中でないか確認します。もし PID が指すプロセスが死んでいれば、古いロックをクリアして続行します。
2.  **ジョブの復旧**: デーモン起動時、`~/.fbqueue/queue/running/` に残っているジョブがあれば、それらを `~/.fbqueue/queue/new/` に移動して再投入します（デーモンがクラッシュした場合の自動復旧）。
3.  **`capacity` の動的読み込み**: `~/.fbqueue/config` から現在の `capacity` 設定を読み込みます。これにより、デーモンを再起動せずにリソース上限を変更できます。
4.  **キャンセル処理**: `~/.fbqueue/queue/cancel/` を監視し、キャンセル指示があるジョブ (ID) を `running_jobs` から特定してプロセスを終了させ、`failed/` に移動します。
5.  **終了ジョブの回収**: `running_jobs` リストを巡回し、完了した子プロセスを `try_wait()` で回収します。終了コードに応じて `done/` または `failed/` に移動します。
6.  **新規ジョブの投入**:
    *   現在の `used_capacity` を計算。
    *   `used_capacity` が `capacity` 未満であれば、`~/.fbqueue/queue/new/` から最も古い (ID の小さい) ジョブを一つ選択。
    *   そのジョブの `cost` が利用可能な `capacity` に収まる場合、`running/` に移動し、子プロセスとして起動。
    *   標準出力/エラーは、ジョブスクリプトや CLI で指定されたパス、または PBS 互換のデフォルトパス（`job.cwd/job_name.oID`）にリダイレクト。
7.  **オートオフ**: `running_jobs` が空で、かつ `new/` に新規ジョブが 5 分間 (300 秒) なければ、デーモンは `daemon.pid` を削除して自身を終了します。

## 6. リソース管理 (Capacity & Cost)

FBQueue は、CPU コア数に限定されない抽象的なリソース管理モデルを採用しています。

*   **`capacity`**: マシン上で利用できる合計リソースの「許容量」。`~/.fbqueue/config` ファイルに `capacity: N` の形式で記述します (デフォルトは `min(利用可能な論理コア数, 8)`)。この値はデーモンがループのたびに読み込むため、動的な変更が可能です。
*   **`cost`**: 各ジョブが消費するリソースの「コスト」。`fbqueue sub -c N` で指定します (デフォルト: 1)。またはジョブスクリプト内の `#$ -pe smp N` などからも自動的に読み込まれます。

**運用例**:
*   1 CPU コアを使うジョブは `cost: 1`。
*   1 枚の GPU を独占するジョブは `cost: 8` など、ユーザーが独自に重み付けを定義。
*   `capacity` を `16` に設定していれば、`cost: 1` のジョブなら 16 個、`cost: 8` のジョブなら 2 個まで同時に実行されます。

## 7. ジョブスクリプトのプリパース機能

`fbqueue sub <script.sh>` と実行された場合、FBQueue は `script.sh` の先頭 100 行を読み込み、以下のディレクティブを解析してジョブ設定に反映します。

*   **ディレクティブの接頭辞**: `#$`, `#PBS`, `#SBATCH`
*   **サポートするオプション**:
    *   `-N <name>` / `-J <name>`: ジョブ名。
    *   `-c <cost>` / `-n <cost>`: コスト（並列数）。
    *   `-pe smp <cost>`: SGE 形式のコスト指定。
    *   `-o <path>`: 標準出力のリダイレクト先。
    *   `-e <path>`: 標準エラー出力のリダイレクト先。
*   **優先順位**: コマンドラインで明示的に指定されたオプションが、スクリプト内のディレクティブよりも優先されます。

## 8. 開発の歴史と経緯

このツールは、以下のユーザー要求と課題解決のために開発されました。

*   「共用サーバで自分個人が使えるコア数（例: 8コア）を占有し、その枠内で大量の計算ジョブを消化したい」
*   「既存のジョブスケジューラがない環境や、あったとしても自分のキューで埋め尽くしたくない」
*   「シェルスクリプトの不安定さを解消し、確実にジョブを回したい」
*   「Rust 製で依存ライブラリなし。あらゆるサーバで動く単一バイナリ」
*   「Windows ネイティブでも動作し、Mac は不要」
*   「ポートを使わないファイルベースの通信と管理」
*   「ユーザーが理解しやすい 1 から始まる連番ジョブ ID」
*   「主要ジョブスケジューラ（SGE/PBS/Slurm）のコマンドエイリアスやスクリプト互換」

これらの要件を満たすため、`fbq` として実装を開始しましたが、Facebook Pixel の `fbq` との混同を避けるため `fbqueue` へと名称を変更しました。

## 9. 今後の課題 (ToDo / Roadmap)

本マニュアル公開時点での主要な ToDo は以下の通りです。

*   **バックグラウンド・アーカイブ**: アイドル時間を利用して、古い完了ジョブとログファイルを `tar.gz` に圧縮・整理する機能。i-node の消費抑制とディスク容量節約に貢献します。
*   **ディレクトリ探索ロジックの柔軟化**:
    *   `FBQUEUE_DIR` 環境変数を導入し、データディレクトリの場所を明示的に指定できるようにする。
    *   カレントディレクトリから親ディレクトリへ遡り、`.fbqueue` ディレクトリを探す機能（プロジェクトごとのキュー管理を可能にする）。
*   **動的なリソース変更**: デーモンを停止させることなく、`~/.fbqueue/config` ファイルの `capacity` 変更をリアルタイムで反映させる機能の強化。
*   **成功/失敗のサマリー表示**: `fbqueue stat` コマンドで、完了/失敗したジョブの終了コードや簡単な統計情報を表示する機能。
*   **ジョブ依存関係**: `#$ -hold_jid <JOB_ID>` のような形式で、前のジョブの完了を待って次のジョブを実行する機能。
